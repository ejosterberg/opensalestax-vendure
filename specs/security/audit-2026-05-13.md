## OWASP A01-A10 manual review

### A01 — Broken access control

- ✅ **Trusted-input boundary.** Plugin reads only `OpenSalesTaxPlugin.init()` arguments + `process.env`. No filesystem reads, no network listens, no config-file parsing.
- ✅ **No inbound HTTP routes.** `grep -rn 'Controller\|Resolver\|@Get\|@Post' src/` → zero matches in source. (`src/lib/ostax-client.ts:125` matches "controller" as part of `AbortController` — N/A.)
- ✅ **No debug endpoints.** No `app.get(`, `app.post(`, `app.listen` in source.

### A02 — Cryptographic failures

- ✅ **No hardcoded credentials.** `grep -rn 'eyJ\|sk_\|pk_\|Bearer ' src/` → zero matches.
- ✅ **No TLS verification disabled.** No `rejectUnauthorized: false`, no `NODE_TLS_REJECT_UNAUTHORIZED` references.
- ✅ **HTTP plaintext warning emitted.** `OstaxTaxLineStrategy.init()` logs a `Logger.warn` if `apiUrl` starts with `http://`, calling out the plaintext-in-production case explicitly.
- ✅ **API token never logged.** `apiToken` is captured in `LoadedConfig` and passed to `OpenSalesTaxClient` only as the `X-API-Key` header. No `Logger.*` references include it.

### A03 — Injection

- ✅ **JSON request bodies, no string concatenation.** `OpenSalesTaxClient.request()` uses `JSON.stringify(body)`, never template-string interpolation of user input into the body.
- ✅ **URL not built from runtime input.** `baseUrl` is captured at construction (validated). The path component is always one of `/v1/calculate` or `/v1/health`, hardcoded.
- ✅ **No `eval` / `new Function` / `child_process`.** `grep` confirms.
- ✅ **Logs are structured.** `Logger.*` calls use template strings, but interpolated values are limited to: country code (`'US'`/`'CA'`), engine status, error messages — never user input.

### A04 — Insecure design

- ✅ **Fail-soft default tested.** `OstaxTaxLineStrategy` returns `[]` + `Logger.warn` on engine error by default. Verified by `tests/unit/ostax-tax-line.strategy.test.ts` "fail-soft" cases.
- ✅ **Fail-hard mode tested.** `OSTAX_FAIL_HARD=1` / `failHard: true` makes engine errors throw. Verified by "fail-hard" tests.
- ✅ **Non-USD / non-US short-circuits.** Strategy returns `[]` without making any HTTP call when `currencyCode !== 'USD'` or `countryCode !== 'US'` or ZIP fails regex. Verified by "gating" tests.
- ✅ **No silent failures.** Every `catch` either logs a warning (fail-soft) or throws (fail-hard). No bare `catch {}`.

### A05 — Security misconfiguration

- ✅ **No HTTP surface added by the plugin.** Re-verified by grep audit at A01.
- ✅ **No Dockerfile shipped** (the plugin is an npm dep; the merchant runs Vendure in their own image). `ls Dockerfile docker-compose.yml` returns "No such file."
- ✅ **No `NODE_ENV=development` assumption baked in.** Source has zero references to `NODE_ENV`.
- ✅ **No filesystem writes.** `grep -rn 'fs\.write\|fs\.append' src/` → zero matches. Logging goes through Vendure's `Logger` (the merchant's configured sink), never to a plugin-owned file.

### A06 — Vulnerable & outdated components

- ✅ **`npm audit --omit=dev --audit-level=high`: 0 vulnerabilities.**
- ⚠️ **Dev-dep audit shows 6 (5 moderate, 1 high).** All are inside `@vendure/core`'s deep transitive tree (apollo-server v4 EOL, lodash via `@nestjs/graphql`). These ship to **us** as devDeps, not to merchants — `@vendure/core` is a `peerDependency` in the published `package.json`. Disposition: **accept and track upstream.** A future Vendure release with `@nestjs/graphql` patched will silently fix it on our side; opening a tracking issue here would be churn since we don't control the fix.
- ✅ **`@vendure/core` is a `peerDependency`,** not a direct dep — the merchant's install picks up their own pinned (and presumably hardened) version.
- ✅ **CI runs the audit.** `.github/workflows/ci.yml` has a step that runs `npm run audit` (`npm audit --omit=dev --audit-level=high`) on every push and PR.

### A07 — Identification & authentication failures

- **N/A.** Documented reason: the plugin exposes no authentication surface. There is no inbound HTTP, no JWT, no session handling, no credential storage. The trust boundary is the merchant's Vendure process (constitution §7). All the categories that A07 typically covers — login flows, session fixation, credential rotation, MFA — do not apply because there are no credentials and no logins.

### A08 — Software & data integrity failures

- ✅ **`package-lock.json` committed** at the initial scaffold commit and updated in lock-step with each dep change.
- ✅ **CI runs `npm ci`** (lockfile-strict), not `npm install`. See `.github/workflows/ci.yml`.
- ✅ **No `postinstall` scripts.** `grep -E '"postinstall"' package.json` → no match.
- ✅ **`prepublishOnly` runs the full check + build** before publishing — prevents shipping unbuilt or untested artifacts.
- ✅ **`package.json` `files` field allowlists** `['dist', 'README.md', 'LICENSE', 'CHANGELOG.md', 'SECURITY.md']`. Verified by `npm pack --dry-run`: 16 files in the tarball; no `src/`, no `tests/`, no `.env*`, no `node_modules/`.

### A09 — Security logging & monitoring failures

- ✅ **No PII in logs.** Strategy logs include only: country code (e.g. `'US'`), engine status (`status`, `version`, `db`), HTTP status codes, error messages from caught exceptions. `grep -rn 'shippingAddress\|customerEmail\|streetLine\|productVariant\.name' src/` → only matches are in JSDoc comments, not logged values.
- ✅ **Engine errors logged at WARN/ERROR with status + message.** The status-code surface for engine errors is captured by `OpenSalesTaxApiError.status`.
- ✅ **Uses Vendure's `Logger`** (merchant-configured sink) — no plugin-owned file logger.
- ✅ **Audit log review.** Manually inspected `Logger.*` call sites in `src/strategies/ostax-tax-line.strategy.ts` (lines 75-94, 100-145). Every interpolated value reviewed; none contain customer addresses (beyond country code), product names, or order details. The previous build had a `zip=${zip5}` debug log; **fixed in this audit pass** — replaced with a non-PII message.

### A10 — Server-side request forgery (SSRF)

- ✅ **`apiUrl` validated at plugin init.** `loadConfig()` requires the URL to parse and pass a scheme allowlist `['http:', 'https:']`. Same allowlist is enforced again at `OpenSalesTaxClient` construction (defense in depth).
- ✅ **No request-time URL interpolation.** The base URL is captured once at construction; the only dynamic component is the request path, which is hardcoded to `/v1/calculate` or `/v1/health`.
- ✅ **No redirect-following to other hosts.** Node's `fetch` (via undici) does not follow cross-origin redirects by default. Confirmed by reading the implementation in `src/lib/ostax-client.ts:request()`: no `redirect: 'follow'` / `redirect: 'manual'` override is set, and we do not implement our own redirect logic.
