## OWASP A01-A10 delta vs v1.0.0

The v1.0.0 audit (`audit-2026-05-13.md`) walked all 10
categories. v1.1.0 changes:

- **A01** Broken access control — N/A delta. New strategy
  reads only `RequestContext`, `Zone[]`, `Channel`, and
  optional `Order` — all Vendure-supplied. No new external
  inputs.
- **A02** Cryptographic failures — N/A delta. No new keys,
  tokens, or TLS surface.
- **A03** Injection — N/A delta. No new HTTP calls, no string
  interpolation of user input, no `eval`.
- **A04** Insecure design — N/A delta. Fail-soft behavior
  unchanged. New strategy adds a fallback path (no US zone
  found → channel default) which is *more* graceful than v1.0.
- **A05** Security misconfiguration — N/A delta. No new
  routes, ports, or filesystem writes.
- **A06** Vulnerable & outdated components — Coverage: dependabot
  config added (`.github/dependabot.yml`) for weekly npm + GitHub
  Actions updates, grouped to keep PR noise low.
- **A07** Identification & authentication failures — N/A
  (still no inbound auth surface).
- **A08** Software & data integrity failures — N/A delta.
- **A09** Security logging & monitoring failures — Re-checked.
  The new strategy logs only a one-shot WARN containing static
  text + no PII. The fallback path doesn't include any address
  or order data.
- **A10** SSRF — N/A delta (no HTTP calls in the zone strategy).

## OWASP A01-A10 delta vs phase-02

- **A01-A10**: no new findings. The new options
  (`categoryByTaxCategoryName`, `defaultCategory`) are
  validated at `loadConfig()` time against a fixed allowlist
  — invalid values throw with a clear error before the engine
  is ever called. The mapping is frozen on `LoadedConfig` so
  the strategy can trust it. The OST `category` value sent on
  the wire is one of the 7 allowlisted values, never
  user-controlled input.
